## Pending decisions (locked 2026-06-04)

| D-18 | Challenge Pool Architecture: Option B — Concept Pool. All challenges for a given Concept are owned by that Concept at platform level (L0). Enemies, BossPhases, and MiniChallenges reference the pool by Concept ID + difficulty filter. Challenges are not embedded in their users. Enables cross-campaign reuse; prevents duplication across stages. | Locked | 2026-06-04 | Human approved. ForgeMinds is a platform — same concept appears in multiple campaigns. Option B is the only model that supports this without duplication. |
| D-19 | Authoring Format: YAML. All ForgeMinds content files (campaign.yaml, stage files, cast files) use YAML as the serialisation format. Most readable for non-engineer content authors; best tooling ecosystem (VS Code schema validation, linting, AI generation output). | Locked | 2026-06-04 | Human approved. |
| D-20 | Game Runtime: Phaser + React Hybrid. Phaser 3 handles the game canvas (scene management, tilemaps, sprites, camera, input, animation). React handles the application shell, UI overlays, HUD, and all tooling (Blueprint Viewer, future Content Studio). Beat Controller layer (React-side) drives Phaser from the content model — Phaser never contains Beat-type logic. | Locked | 2026-06-04 | Human approved. Both already installed; proven pattern for web-based JRPGs; Phaser for game world, React for everything else. |
| D-20A | Application Structure: Single Next.js Application. One app hosts game, Blueprint Viewer, and future tooling. Code splitting via `next/dynamic` (ssr: false) keeps Phaser out of non-game routes. Designed to be splittable into separate apps in future if needed. | Locked | 2026-06-04 | Human approved. Blueprint Viewer already lives here; operational simplicity outweighs split-app benefits at v1 scope. |
| D-21 | React Integration Strategy: React Shell + Phaser Canvas. Phaser mounts inside a `PhaserGame.tsx` React component targeting a `<canvas>` element. React owns all DOM UI (menus, dialogue overlays, HUD, settings). A thin `GameEventBus` bridges Phaser ↔ React for cross-boundary events. When React modals are active, Phaser input is suspended. | Locked | 2026-06-04 | Human approved. Cleanest separation of concerns; React for UI/tooling, Phaser for game world rendering and logic. |
| D-22 | YAML Loading Strategy: Hybrid — runtime parsing in development, build-time compilation in production. Dev: YAML parsed in-browser via js-yaml; hot-reload friendly; fast content iteration. Prod: YAML compiled to TypeScript modules at build time; Zod validation; optimised content bundles; no runtime parser. Reason: ForgeMinds is content-heavy — authoring speed during campaign creation outweighs production parsing performance. | Locked | 2026-06-04 | Human approved. Hybrid preserves both dev DX (instant hot-reload) and production integrity (type safety, validation, no parser bundle). |
| D-23 | State Management Boundaries: Three categories. (1) Frame/runtime state → Phaser internal (character position, animation, camera, NPC movement — never persisted, never queried by React). (2) Session state → Zustand `sessionState` slice (current Beat, active HP, current dialogue state, stage-local flags — resets on stage load). (3) Persistent progress state → separate persistence layer / Zustand `progressState` slice (ConceptMastery, quest completion, inventory, XP, levels, campaign flags — survives death and session end). | Locked | 2026-06-04 | Human approved. Three clean categories prevent frame-state pollution of progress state; Zustand owns what React needs; Phaser owns what the game loop needs. |
| D-24A | Beat Controller Ownership: React-side. The BeatController is a React module (hook or service class) that reads the Campaign model's Beat sequence and drives execution. It fires commands to Phaser; it receives completion events from Phaser via the GameEventBus. Phaser is a "dumb renderer" — it executes world actions but never contains Beat-type logic or campaign progression decisions. | Locked | 2026-06-04 | Human approved. Content orchestration must remain independent of Phaser. Phaser executes world actions; React owns campaign progression and Beat execution. |
| D-24B | Scene Management: One Stage Runtime Scene pattern. Three scenes only: BootScene (engine init), PreloaderScene (asset loading), StageRuntimeScene (all stage gameplay). StageRuntimeScene is reusable — it loads stage content dynamically when a new Stage begins. One scene per stage conceptually; one reusable scene technically. No hardcoded per-stage Phaser scenes. | Locked | 2026-06-04 | Human approved. Stage = concept; StageRuntimeScene = technical implementation. Clean separation between content identity and engine execution. |
| D-24C | World Reality Ownership: 5 rules. (WR-1) Phaser owns all spatial reality — position, velocity, collision, animation. (WR-2) Zustand owns all progress reality — mastery, quests, XP, levels. (WR-3) No Phaser scene subscribes to Zustand. (WR-4) No React component reads Phaser internal state. (WR-5) All Phaser ↔ React crossings are via GameEventBus — no shared mutable objects. | Locked | 2026-06-04 | Established in Phase 3.2. Prevents frame-state pollution of progress state; enforces clean unidirectional data flow. |
| D-25 | Beat Completion Events: Each Beat type has exactly one defined completion condition. EXPLORATION is ambient and does not complete. Completion is signalled by React UI event, GameEventBus event from Phaser, or system callback. No Beat may be considered complete by timeout or polling. | Locked | 2026-06-04 | Established in Phase 3.2. Deterministic completion model required to prevent save/progress errors. |
| D-26 | Scene Transition Model: Campaign model never moves to Phaser. BeatController derives StageContext from Campaign model and passes it to Phaser at scene start. Phaser sees only rendering commands — never content semantics. PreloaderScene loads assets from manifest; StageRuntimeScene renders from StageContext. | Locked | 2026-06-04 | Established in Phase 3.2. Enforces the "Phaser is a dumb renderer" constraint from D-24A. |
| D-27 | Content Versioning: Version string format `{major}.{minor}.{contentHash[:8]}`. Exact match → load normally. Minor version difference → compatibility mode (stage-completion-based restore, player informed). Major version difference → save incompatible. Major/minor set by content author in campaign.yaml; hash appended automatically by pipeline. | Locked | 2026-06-04 | Established in Phase 3.3. Protects players from silent content drift while allowing minor updates. |
| D-28 | Content Loading Pipeline — Seven stages: (1) Content Discovery → file manifest. (2) Validation → parse + schema + structural rules + cross-file pre-check. (3) Reference Resolution → ID refs replaced with objects, bidirectional links built. (4) Challenge Pool Resolution → pre-filtered Challenge arrays per concept/difficulty, deferred to runtime for seeded shuffle. (5) Theme Resolution → ThemeOverrides merged, invariant core verified. (6) Campaign Compilation → single immutable CompiledCampaign object. (7) Runtime Consumption → BeatController + Blueprint Viewer share same object. | Locked | 2026-06-04 | Established in Phase 3.3. |
